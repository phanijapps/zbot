Providers Page: Current vs Proposed

The current page has 6 critical UX issues for non-technical users. Here's what needs to change.

Critical Issues Found

1. No inline editing
API key rotates? Delete and recreate the entire provider. No edit button anywhere.
2. Raw comma-separated models
Users type "gpt-4o, gpt-4o-mini" by hand. No autocomplete, no validation, no way to remove one model.
3. No onboarding guidance
New user sees "Add Provider" with a blank form. No explanation of what a provider is or why they need one.
4. Doesn't use design system
450 lines of inline Tailwind. Ignores .card, .btn, .form-input, .badge, .empty-state classes from components.css.
5. No model context for new users
Tiny wrench/eye/brain icons with no labels. A non-technical user has no idea what they mean.
6. No test-before-save
You create a provider, then separately test it. Bad key? Now you have a broken provider you can't fix.

Which redesign direction do you prefer?

Both fix all 6 issues. The difference is scope and ambition.

A

Polish & Fix (smaller scope)

Keep the split-panel layout. Fix the 6 issues within the existing structure:

  • Add edit mode to detail view (editable API key, base URL, models)
  • Replace comma-input with tag chips (type + enter, click X to remove)
  • Add model capability tooltips with labels ("Tool Calling", "Vision", etc.)
  • Add "Test & Save" button in create flow
  • Better empty state with guided first-provider setup
  • Migrate all inline Tailwind to semantic CSS classes
~1 session of work. Surgical fixes, same layout.
B

Full Redesign (recommended)

Rethink the page for non-technical users who just want agents working:

  • Guided setup wizard — "Connect your first AI provider" with preset cards (logo + name + "Connect" button)
  • Provider cards grid instead of sidebar list — each card shows name, status badge (connected/error), model count, active indicator
  • Slide-over detail panel with editable fields, model tag chips with capability badges + context window
  • Inline connection test with real-time status indicator (green pulse = connected, red = error)
  • Model browser — when adding models, show available models from registry with capability filters
  • Full migration to semantic CSS (`.card`, `.btn`, `.form-input`, `.badge`)
  • Better empty state — explains what providers are, why you need one, shows the 3 most popular presets prominently
~2-3 sessions. Complete UX rethink. Production-quality.
