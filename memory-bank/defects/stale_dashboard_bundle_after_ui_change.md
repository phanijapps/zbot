# Defect — stale UI bundle served on port 18791 after UI changes

## Symptom

Navigating a browser to `http://<host>:18791/` (the zerod dashboard) can
render an old compiled UI bundle while `npm run dev` on port 3000 shows
current code. On mobile devices (phones on the same LAN) the stale
bundle shows a "Connection Failed / gateway error" panel because the
old JS hard-codes `localhost:18791` as the gateway URL and
"localhost" on the phone is the phone itself.

## Reproduction

1. Make any change to `apps/ui/src/**`.
2. Save without running `npm run build`.
3. Open `http://<mac-lan-ip>:18791/` from a phone on the same WiFi.

Result: the page renders, but the bundle it loads (`/assets/index-*.js`)
is whatever was last written to `dist/` — possibly weeks old. Any
behaviour the change was supposed to fix doesn't land. The phone's
gateway-config fallback (dynamic `window.location.hostname`) was
introduced in commit `1359357`; older bundles still have the
`localhost:18791` default and throw on connect.

## Root cause

Zerod's daemon is launched with `--static-dir ./dist`, so port 18791
doubles as the production UI host by serving `dist/index.html` and
`dist/assets/*`. That dist tree is only regenerated by an explicit
`npm run build` — there's no watcher tying it to `apps/ui/src/**`.

`dist/` is in `.gitignore`, so the tree on a fresh clone is empty until
the first build; and nothing in the dev workflow keeps it in sync
after.

Four breaks combine:

1. Dev workflow serves two different UIs (vite on 3000, zerod on 18791)
   but devs usually only drive the 3000 one.
2. Phone testing funnels to 18791 because 3000 is the dev-only origin
   and Vite's HMR module graph is slow on mobile.
3. No build step runs on code change or before zerod starts.
4. Nothing surfaces "this bundle is older than your source" — the old
   hash loads silently.

## Workaround today

After every UI change that needs to be visible on the phone:

```
cd apps/ui && npm run build
```

Hard-reload the phone (pull-to-refresh / long-press reload) to evict
the cached JS chunk.

## Proper fixes to consider

- **Build gate on zerod boot**: if `dist/index.html` is older than the
  latest mtime under `apps/ui/src/`, refuse to start (or emit a loud
  warning) and hint the user to `npm run build`.
- **Auto-rebuild in development**: add `npm run build --watch` as a
  sibling dev process to `vite`, so `dist/` tracks source.
- **Point zerod at vite in dev**: pass `--static-dir` only when a
  release flag is set; otherwise proxy to `http://localhost:3000`.
  Eliminates the dual-UI problem entirely.
- **Version marker**: inject a source-hash into the bundle's title
  attribute so we can eyeball "is this the current code?" on any
  device.

## Files involved

- `apps/daemon/src/main.rs` — CLI parses `--static-dir`.
- `gateway/src/http/mod.rs` — `.nest_service("/", ServeDir::new(...))`.
- `apps/ui/package.json` — `build` script = `tsc && vite build`.
- `apps/ui/src/services/transport/index.ts` — `defaultConfig()`
  dynamic gateway URL (fix landed `1359357`).

## Discovered

2026-04-20 by @phanijapps — phone on LAN hit
`http://192.168.4.23:18791` and saw "Connection Failed" while the
dev-machine desktop at `localhost:3000/chat` was working fine. Bundle
in dist/ was `index-DIW-a3o7.js`; current source compiled to
`index-CFZvo4nc.js` after `npm run build`.
