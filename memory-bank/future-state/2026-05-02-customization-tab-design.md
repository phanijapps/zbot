# Customization Tab — Design

**Status:** Design (awaiting review)
**Date:** 2026-05-02
**Owner:** phanijapps
**Target branch:** `develop`

## Problem

Today, the only way to edit AgentZero's prompt-assembly markdowns (`SOUL.md`, `INSTRUCTIONS.md`, `chat_instructions.md`, `distillation_prompt.md`, `intent_analysis_prompt.md`, the shards under `shards/*.md`) is to open them in a terminal text editor. Users who interact with AgentZero through the web UI have no in-app surface for this. The wizard is a one-time onboarding tool and doesn't expose individual file content.

We add a new **Customization** tab in Settings that lists the markdown files under `<vault>/config/` and lets the user edit and save them, with file-watcher integration so external edits show up live.

## Goals

1. New "Customization" tab in Settings (5th, alongside Providers / General / Logging / Advanced).
2. List all markdown files under `<vault>/config/` (root markdowns + `shards/*.md`).
3. Click a file → load its content in an editor pane → save back to disk.
4. Auto-detect external edits via the existing file watcher and refresh the UI accordingly.
5. Reuse existing CSS classes — no new style definitions.

## Non-goals

- Create / delete operations from the UI (edit-only per design discussion).
- JSON config files (`mcps.json`, `providers.json`, `models.json`, `settings.json`) — they have dedicated UI surfaces elsewhere.
- The `wards/` subdirectory — user data, not config.
- Live markdown preview, syntax highlighting, code-fold, or git history view.
- Multi-file diff or merge resolution beyond a "file changed on disk" banner.
- Multi-user concurrent edit reconciliation (single-user app).

**Three components**:

## Backend API

All new endpoints live in `gateway/src/http/customization.rs` (new file). Routes registered in `gateway/src/http/mod.rs` next to the other settings routes.

### `GET /api/customization/files`

Lists allowed markdown files. Computed by walking `<vault>/config/` and `<vault>/config/shards/`, filtering to the allow-list rule (above).

```ts
type FileEntry = {
  path: string;          // "SOUL.md" or "shards/memory_learning.md"
  kind: "root" | "shard";
  size: number;          // bytes
  modifiedAt: string;    // ISO8601 from mtime
  autoGenerated: boolean; // only true for OS.md
};
type ListResponse = { success: true; files: FileEntry[] };
```

### `GET /api/customization/file?path=<relative>`

Returns file content + version for optimistic concurrency.

```ts
type FileResponse = {
  success: true;
  path: string;
  content: string;
  version: string;        // mtime as ISO8601
  autoGenerated: boolean;
};
```

Errors:
- `400` — path missing, fails allow-list validation, contains `..`, or starts with `/`.
- `404` — path is allow-listed but file doesn't exist.

### `PUT /api/customization/file`

```ts
type SaveRequest = {
  path: string;
  content: string;
  expectedVersion: string;  // last-loaded mtime
};
type SaveResponse = {
  success: true;
  path: string;
  version: string;          // new mtime after write
};
```

Errors:
- `400` — same path validation as GET.
- `409` Conflict — current mtime ≠ `expectedVersion`. Body:
  ```ts
  { success: false; error: "version mismatch"; currentVersion: string; currentContent: string }
  ```
  UI shows the user both versions and asks which wins.
- `413` Payload Too Large — content > 1 MB. Defensive cap.

### Server-side path validation (mandatory)

```rust
fn validate_customization_path(p: &str) -> Result<PathBuf, &'static str> {
    if p.is_empty() || p.starts_with('/') || p.starts_with('\\') {
        return Err("invalid path");
    }
    if p.contains("..") {
        return Err("invalid path");
    }
    if !p.ends_with(".md") {
        return Err("only markdown files allowed");
    }
    let parts: Vec<&str> = p.split('/').collect();
    match parts.as_slice() {
        [_file] => Ok(PathBuf::from(p)),
        ["shards", _file] => Ok(PathBuf::from(p)),
        _ => Err("invalid path"),
    }
}
```

Resolved final path is `vault.config_dir().join(validated_path)`. Server only ever reads or writes that resolved path; never accepts user-supplied absolute paths.

## File watcher integration

The existing `FileWatcher` (`gateway/gateway-services/src/watcher.rs`) already monitors `<vault>/config/`. We add one event type to the existing WebSocket event stream:

```ts
type CustomizationFileChangedEvent = {
  type: "customization_file_changed";
  path: string;          // relative path
  modifiedAt: string;    // new mtime
};
```

The watcher fires the event whenever a `*.md` file in the allow-list changes (created, modified, or deleted), regardless of whether the change came from the daemon itself, an external editor, or `git pull`.

**UI reactions:**

- File is NOT currently selected → refresh the file list (mtime in sidebar updates).
- File IS currently selected AND editor has no unsaved changes → silently re-fetch + update the editor.
- File IS currently selected AND editor has unsaved changes → show a non-modal banner: *"This file changed on disk while you were editing. [View disk version] [Keep editing]"*. User decides.

## State management

`CustomizationTab` owns:

```ts
type State = {
  files: FileEntry[];
  selectedPath: string | null;
  editorContent: string | null;     // null when nothing selected
  diskContent: string | null;       // last-fetched content (for dirty detection + discard)
  loadedVersion: string | null;     // mtime when content was loaded
  isSaving: boolean;
  conflict: { currentContent: string; currentVersion: string } | null;
};
```

Dirty detection: `editorContent !== diskContent`. Discard: `setEditorContent(diskContent)`. Save: PUT with `expectedVersion: loadedVersion`. On 409: populate `conflict` to render the banner.

WebSocket event handler updates `files` immediately on any `customization_file_changed`. If the changed file matches `selectedPath`, branches as described in "UI reactions" above.

## Save flow

1. User edits the textarea → `editorContent` diverges from `diskContent` → "Modified" badge appears, Save button enables.
2. User clicks **Save** → PUT request with `{ path, content: editorContent, expectedVersion: loadedVersion }`.
3. **200 OK** → update `diskContent = editorContent`, `loadedVersion = response.version`. Badge clears.
4. **409 Conflict** → set `conflict` state, render banner. User clicks one of:
    - "View disk version" → `editorContent = currentContent`, `loadedVersion = currentVersion`, `conflict = null`. Their unsaved edits are lost (they chose).
    - "Keep editing" → `loadedVersion = currentVersion`, `conflict = null`. Next save will succeed (overwrites disk).
5. **Other errors** → toast / banner with the server's `error` string.

## Out of scope (documented as future work)

- **Create / delete files from the UI.** Edit-only per design call. If users need new shards, they create them via terminal; they auto-appear in the list (file watcher).
- **Live markdown preview / syntax highlighting / Monaco editor.** Plain textarea suffices; richer surfaces can be added later.
- **Concurrent multi-user edit reconciliation.** Single-user app; not relevant.
- **Diff view between disk and editor versions.** The conflict banner shows just the server's "current content" — user eyeballs the difference. Could add a side-by-side diff later if the workflow demands it.
- **Per-file metadata (description, tags, last-edited-by).** Filesystem mtime is the only metadata.
- **Audit log of edits.** Out of scope; git already tracks user vault commits if they version-control their config.
