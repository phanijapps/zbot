01 / 16
02 Thesis

The thesis

A useful agent memory is
not a key-value cache.

It is a stack of interpretive layers. Each answers a different kind of question. Each can be built independently. The whole composes into an agent that learns.

A minimum-viable implementation can ship Layer 1 alone and the agent will already remember things. The other six layers compose on top, in roughly the order they appear here.

The agent asksAnswered by
"What did I literally observe?"L1 episodic + facts
"What real-world things does this concern?"L2 knowledge graph
"What was true when?"L3 bi-temporal
"What's the bigger concept this belongs to?"L4 hierarchy
"What do I currently believe?"L5 belief network
"How have I done this before?"L6 procedures
"What patterns emerge that I should formalise?"L7 reflective synthesis

05 Layer 1 · facts

Layer 1 — atomic facts

Every fact is a quad,
upserted by (scope, ward, key).

Shape

// memory_facts row
{
  id:              "fact-{uuid}",
  agent_id:        "root",
  ward_id:         "__global__",
  scope:           "global",   // agent | global | ward
  category:        "correction",
  key:             "git.never-pull-shared",
  content:         "Always rebase on shared branches.",
  confidence:      0.85,
  mention_count:   3,
  valid_from:      "2026-01-12T...",
  valid_until:    null,            // L3 bi-temporal
  superseded_by:   null,
  contradicted_by: null,
  epistemic_class: "current",    // current|historical|hypothetical
  source_episode_id: "ep-...",
  pinned:          0,
}

Category weights (default)

categoryweightrole
schema1.6×principles abstracted from corrections
belief1.5×L5 synthesised positions
correction1.5×"never do X / always Y"
strategy1.4×known-good approaches
user1.3×preferences / context
instruction1.2×standing directions
domain1.0×baseline domain knowledge
pattern0.9×observations not yet a strategy
ward · skill · agent0.7-0.8×indexer-managed

Each fact is also indexed by FTS5 (BM25) and a vec0 ANN index. Triggers keep both in sync on insert / update / delete.

06 Layer 2 · knowledge graph

Layer 2 — knowledge graph

Typed entities,
typed edges,
2-hop expansion.

Generic recursive-CTE traversal. max_hops = 2, hop_decay = 0.6, entities below min_kg_confidence = 0.1 are filtered before scoring.

Tables

  • ENT
    kg_entities — id, agent_id, type, name, normalized_hash (dedup), confidence, layer (L4), parent_cluster_id
  • REL
    kg_relationships — directed edges, UNIQUE(src, tgt, type), layer, is_inter_cluster (L4)
  • ALI
    kg_aliases — surface form → canonical entity. Alias resolution during distillation.
  • CAU
    kg_causal_edges — cause → effect observations. Input to procedure mining.
  • EP
    kg_episodes — ingest queue (pending → processing → done/failed). Multiple workers safe.
  • VEC
    kg_name_index — vec0 ANN over entity names. Powers fuzzy entity match.
  • AUD
    kg_compactions — audit trail. Every merge / supersede / prune recorded for undo.

07 Layer 3 · bi-temporal

Layer 3 — bi-temporal

Two timelines.
Supersession over delete.

Fact A is not deleted — its truth interval is closed at the moment we learned otherwise. A query with as_of = 2022-06 still returns fact A.

Columns

columnmeaning
created_at / updated_attransaction time — when learned
valid_fromwhen the fact started being true
valid_untilwhen it stopped (NULL = currently valid)
superseded_byFK to the replacement row
contradicted_bysoft flag; ranks lower but stays visible
epistemic_classcurrent / historical / hypothetical

Recall API

recall_facts_prioritized(
  agent_id, query, limit,
  as_of?: DateTime   // point-in-time
)

Same columns live on kg_entities, kg_relationships, and kg_beliefs — bi-temporal is a cross-cutting concern.

08 Layer 4 · hierarchy · LeanRAG

Layer 4 — hierarchical memory

K-means clusters
into concept aggregates.
Recall walks down
from the LCA.

Builder algorithm

  • 01
    Fetch layer-N entities + embeddings
  • 02
    K-means cluster by cosine — target size 20
  • 03
    Per cluster: LLM aggregate {name, description} (singletons skip LLM)
  • 04
    Write aggregate as layer-(N+1) entity; set parent_cluster_id
  • 05
    For each aggregate pair with λ ≥ 3 underlying edges, LLM-synthesise relation type
  • 06
    Write is_inter_cluster = 1 edge at layer N+1
  • 07
    Repeat until max_layers or sparsity stop

LCA-bounded recall

Map seed entities → their layer-(N+1) clusters → lowest common ancestor. Walk down from LCA for primary results. Surface inter-cluster relations as a separate band (the agent sees "this connects to OTHER concepts: X, Y" without them leaking into the primary list).

Opt-in via execution.memory.hierarchy.enabled.

09 Layer 5 · belief network

Layer 5 — belief network

One belief per subject.
Recency-weighted across
many facts.
Contradictions are first-class.

Synthesis

recency-weight w = 1 / (1 + age_days / 90)

Contradiction graph

Four LLM decisions: logical_contradiction, tension, compatible, duplicate. Only logical and tension are persisted; compatible/duplicate are dropped. Budget: ~20 LLM calls per cycle.

10 Layer 6 · procedures

Layer 6 — procedures

Successful sequences
become replayable.

// procedures row · `steps` is JSON
{
  name: "fetch_and_summarise",
  description: "Fetch a URL and produce a summary.",
  parameters: ["url"],
  steps: [
    {
      action: "shell",
      args:   { command: "curl -sS {{url}}" },
      binds:  ["stdout"]
    },
    {
      action: "llm_summarise",
      args:   {
        text:     "{{step_0.stdout}}",
        max_words: 100
      },
      binds:  ["summary"]
    }
  ],
  success_count: 2,    // primed at 2 by sleep-time miner
  failure_count: 0,
  avg_duration_ms: 1842
}

Mining (sleep-time)

  • 01
    Find pairs of successful sessions with task-summary cosine ≥ 0.82
  • 02
    Pairs must share a tool-call prefix of length ≥ 3
  • 03
    LLM generalises → parameterised procedure (budget ~5/cycle)
  • 04
    Every step's action must be in the tool whitelist
  • 05
    Existing procedures with success_count ≥ 2 are not overwritten

Dispatch — run_procedure(name, params)

Iterates steps, interpolates {{step_N.field}} and {{param}}, increments success_count / failure_count + duration / token cost stats. Procedures are vector-indexed (procedures_index) so recall surfaces "you have a procedure for this."

11 Layer 7 · reflective synthesis

Layer 7 — sleep-time pipeline

Twelve stages. Conservative by design.
One stage failing never aborts the cycle.

Fires on a configurable interval (default 60 min) or on-demand via POST /api/memory/consolidate. All LLM calls use temperature = 0.0 for determinism (handoff is the lone exception at 0.2).

stage 01
Compactor
Entity dedup via cosine + pairwise verifier (defaults deny on LLM fail)
stage 02
Hierarchy builder
K-means clusters → aggregates → inter-cluster edges
stage 03
Synthesiser
Cross-session strategies; dedup at cosine ≥ 0.88
stage 04
Pattern extractor
Successful sessions → parameterised procedures
stage 05
Decay engine
Confidence ÷ (1 + age/half_life); skip recent
stage 06
Contradiction propagation
Fact-drop → KG decay + belief stale
stage 07
Pruner
Below floor → archived; never destroyed
stage 08
Orphan archiver
Zero-edge zero-recall entities
stage 09
Corrections abstractor
3+ corrections → schema principle
stage 10
Conflict resolver
Schema vs schema → supersession + propagation
stage 11
Belief synthesiser
Facts grouped by subject → one belief
stage 12
Belief contradiction detector
Pairwise within subject-prefix neighbourhoods

handoff writer Runs per-session, not per-cycle. 3-5 sentence summary of the just-completed session, written to memory_facts with agent_id = "__handoff__". Injected at the next session's bootstrap.

12 Recall

Recall — the engine

Reciprocal Rank Fusion,
then nine multipliers.

1 · Hybrid retrieval (RRF)

K_retrieval = 3 · budget
RRF_K = 60

for rank, item in fts_results:
  score[item] += 1 / (RRF_K + rank)
for rank, item in vec_results:
  score[item] += 1 / (RRF_K + rank)
// items in BOTH lists get summed → "hybrid"

RRF is robust to score-scale mismatches between lexical (BM25) and vector (cosine) engines — no weight tuning required. The "0.7 × vector + 0.3 × BM25" formula in older docs is a fallback path.

2 · Priority multipliers

multiplierformula / value
category weight1.6 schema · 1.5 belief · 1.5 correction · 1.4 strategy · ...
ward affinity1.3× if matches current ward
temporal decay1 / (1 + age_days / half_life)
mention boost1 + log2(max(mentions, 1))
contradiction penalty0.7× if contradicted_by IS NOT NULL
supersession penalty0 if historical; 0.5 if hypothetical
entity confidencefilter < 0.1 in graph-ANN path
predictive boost1.3× when recalled in past successful sessions

Final results sorted descending, truncated to budget, dropped below min_score = 0.3.

When memory fires

Six loops during
the agent lifecycle.

1
System recall
first user message of a session
system msg @ index 0
2
Intent + memory
before intent-analysis LLM call
classifier prompt
3
Subagent priming
at delegation spawn
child initial history
4
Mid-session refresh
every 5 turns (novelty-filtered)
working-memory mw
5
Post-session distillation
session end
writes L1 + L2
6
Sleep-time pipeline
interval tick · on-demand trigger
writes L1 / L2 / L5 / L6

Loops 1-4 are reads. Loops 5-6 are writes. All six share the same recall_unified(agent_id, query, ward_id, goals, budget, as_of?) entry point on the read side. Each loop has a stable integration surface that can be substituted with a fixture for testing.

14 Per-task LLM routing

Routing

Different jobs deserve
different models.

Tagged tasks

tagcall sites
sleep_timesynth, beliefs, abstraction, conflict, verifier, pattern extractor, handoff (10 sites)
curatorward consolidation phase C
intent_analysisper-prompt routing classifier
distillationpost-session extractor
multimodalvision analysis

Untagged callers (chat, recall) fall through to the orchestrator. Routing is decided at call time by reading execution settings — no daemon restart needed.

Per-ward LLM config (wards/{name}/config.yaml) is a separate override path that applies to ward delegations only.

A reimplementation roadmap

Ten milestones.
Each is independently shippable.

01
MVP
L1 facts, lexical index, save/recall
02
Scoring + vector
vec0, RRF, scoring multipliers
03
Knowledge graph
L2 entities + edges + traversal
04
Bi-temporal
L3 valid_from / supersession / as_of
05
Procedures
L6 steps, interpolation, run_procedure
06
Min sleep-time
L7 compactor, decay, pruner, archive
07
Reflection
corrections abstractor + conflict resolver + pattern miner
08
Hierarchy
L4 clusters, LCA-bounded recall
09
Belief network
L5 synthesis + contradiction graph
10
Polish
handoff, mid-session, predictive boost, observability

A minimum-viable implementation is milestones 1-2 only — the agent already remembers across sessions, with relevance-ranked recall. Everything above that is a force multiplier: graph connectivity (3), truth over time (4), replayable runbooks (5), self-healing memory (6-7), scale (8), and coherence (9).
