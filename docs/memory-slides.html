// agentzero / memory subsystem

How zbot
remembers.

flat fragments + knowledge graph + a recall pipeline that decides what the agent reads next

storage●recall●rescore●graph●lifecycle
01  /  the metaphor

A transcript is not memory.

Every session is a page in the log — captured verbatim, exhaustive, inert.

Memory is the reference book the agent writes from those pages. It's smaller, curated, re-read at the start of every new session.

conversations.db = the log
knowledge.db = the reference book

02  /  two views of memory

Two structures.
One memory.

Flat fragments answer "what should I remember about this?" Self-contained notes with a category, content, confidence, embedding, and lifecycle.

The knowledge graph answers "what is connected to what?" Entities and typed edges, each with their own confidence and last-seen timestamp.

They're linked by source_episode_ids — provenance back to the conversation that produced both. Recall matches a fragment, then walks the graph outward to find related ones.

Fragments carry the content. The graph carries the shape.

03  /  fragment types

Every fragment
has a category.

Categories control how aggressively recall promotes a fragment. They're applied as multipliers after raw search returns.

correction — what the user told you to do
schema — distilled rules from multiple corrections
strategy — cross-session approaches that worked
user — durable facts about the user
instruction — durable directives, conventions
pattern — abstracted tool-call sequences

Plus domain, ward, skill, agent, handoff — scope and sentinel categories.

Each fragment is also in one of four lifecycle states: active, contradicted, superseded, archived.

04  /  the knowledge graph

Entities and edges,
each with their own life.

Both ends of an edge and the edge itself carry independent confidence and last_seen_at. None is renewed on recall — only when the entity actually shows up in a new conversation.

Confidence decays exponentially against last_seen_at, with a default half-life of 90 days for both entities and relationships. Floored at min_confidence = 0.01; rows touched within 24h are skipped.

Orphan entities (no relationships, old last_seen_at) are flagged and archived. Archived rows are never deleted — just hidden from recall.

recall_config.json → kg_decay
entity_half_life_days: 90 · relationship_half_life_days: 90 · min_confidence: 0.01
05  /  storage

Two SQLite files
do all the work.

knowledge.db is the reference book: typed fragments, KG entities and relationships, episode chains, compaction audit.

conversations.db is the daily log: chat history. Separate file, separate concerns — memory never touches it directly. It goes through the ConversationStore trait.

Embeddings live in their own sqlite-vec virtual table, joined to facts by fact_id. FTS5 indexes the same content for keyword search. The hybrid recall pipeline reads both.

stored under $XDG_DATA_HOME/agentzero/ on Linux
equivalent paths on macOS / Windows
06  /  recall — finding candidates
recall_config.json — vector_weight: 0.7 · bm25_weight: 0.3 · candidates returned: limit × 2 · rescore is where the actual top-K is picked.
07  /  rescore — adjusting the scores
[1] CATEGORY WEIGHT schema 1.6 / correction 1.5 / strategy 1.4 … × weight
[2] CONTRADICTION PENALTY contradicted_by is set × 0.7
[3] TEMPORAL DECAY valid_until set · class-aware × 0.1–0.3
[4] WARD AFFINITY candidate's ward matches active ward × 1.3
[5] SUPERSESSION FILTER superseded_by is set DROP
[6] MIN-SCORE FLOOR noise gate · keeps short queries clean DROP < 0.3
[7] SORT & TRUNCATE top-K by final score max_facts 10

Raw scores meet
opinionated rules.

Rescore runs client-side after SQL returns. Category preference, contradiction penalty, supersession exclusion, and the noise floor all live here — not buried in a SQL view.

Schemas outrank corrections because they're distilled. Skill and agent indices sit at the bottom because they exist for lookup, not for behavior shaping.

Lime steps multiply scores upward; coral steps demote or drop. A separate max_recall_tokens = 3000 cap limits the total size of the formatted block.

recall_config.json — every multiplier tunable.
min_score: 0.3 · contradiction_penalty: 0.7 · ward_affinity_boost: 1.3 · max_facts: 10
08  /  graph traversal

After rescore,
walk the graph.

For each top fragment, follow source_episode_ids to the entities it referenced. From those, walk outward along kg_relationships.

Each hop multiplies relevance by hop_decay = 0.6. Direct neighbors weigh 0.6×; two hops out weigh 0.36×. Capped at max_graph_facts = 5.

A query about commit hygiene matches one correction; the traversal pulls in git workflow, code review, release process — even though none of those words appeared in the query.

recall_config.json → graph_traversal
max_hops: 2 · hop_decay: 0.6 · max_graph_facts: 5
09  /  session lifecycle
ward-scoped: the handoff block is skipped if the previous session was in a different ward · prevents bleed-through.
always-on injection: active corrections + active goals don't rely on recall — they're pulled directly, every session.
10  /  background maintenance

Eight jobs,
each does one thing.

An hourly worker runs the cycle. Jobs are independent — one failing doesn't stop the others.

The LLM-using jobs have their own throttles so the cycle doesn't burn tokens on jobs that have no new material to chew on. Default cadence: 24h for Compactor, Synthesizer, PatternExtractor, CorrectionsAbstractor, and ConflictResolver.

The cheap mechanical jobs — DecayEngine, OrphanArchiver, Pruner — run every cycle. Decay touches last_seen_at; orphan check + prune walks the result.

settings.json → execution.memory
correctionsAbstractorIntervalHours · default 24
conflictResolverIntervalHours · default 24
11  /  configurability
two files

Every knob exposed. Sane defaults.

settings.json → execution.memory
background-job throttles 
{
  "execution": {
    "memory": {
      "correctionsAbstractorIntervalHours": 24,
      "conflictResolverIntervalHours": 24
    }
  }
}

Set either to 0 to fire on every hourly cycle. Missing keys fall back to compiled defaults. A corrupted file logs a warning and uses defaults.

Partial files deep-merge with defaults — user values win per key.

12  /  the closed loop

From correction
to distilled rule.

A single user correction enters the system. The next session sees it in ## Active Corrections regardless of whether anyone queried for it.

Across multiple sessions it accumulates with siblings. The CorrectionsAbstractor distills 3+ corrections that share a theme into a single schema fragment with weight 1.6 — which outranks the raw 1.5 corrections.

If two distilled rules ever disagree, the ConflictResolver finds the pair by cosine similarity, asks an LLM judge, picks the higher-confidence winner. The loser gets superseded_by set and disappears from recall.

Raw feedback becomes principles · principles get arbitrated · stale ones disappear. No one curated this notebook by hand.
