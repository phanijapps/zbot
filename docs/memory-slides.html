01  /  the metaphor

A transcript is not memory.

Every session is a page in the log — captured verbatim.

Memory is the reference book the agent writes from those pages. It's smaller, indexed, and the only thing read at the start of the next session.

every session = a page
memory = the reference book

02  /  the fragments

Memory is many small fragments,
not one big blob.

Each fragment carries a category (correction, schema, strategy, user, handoff, goal…), a content body, an embedding, a confidence score, and a short opaque key.

Recall doesn't replay history — it pulls the handful of cards that matter for the current turn, ranked by similarity then multiplied by category weight.

A correction is what you told it.
A schema is what it learned.
A strategy is what worked.
A user fact is what's true about you.
A handoff is what just ended.
A goal is what's in flight.

03  /  architecture

Everything memory-related
lives in one crate.

Before extraction, memory code was scattered across five files in three crates. Adding a new component meant touching the gateway, the execution crate, and services together.

Now gateway-memory owns it all: sleep components, recall pipeline, LLM factory abstraction, and the construction factory itself.

Trait boundaries keep it decoupled. MemoryLlmFactory hides providers; MemoryFactStore & siblings hide the SQLite backend. The execution crate composes — memory produces blocks, the runtime decides where they go.

04  /  storage

Two SQLite files
do all the work.

knowledge.db is the reference book: typed fragments, KG entities, KG relationships, episode chains, compaction audit.

conversations.db is the daily log: chat history. Separate file, separate concerns — memory never reads it directly, only the ConversationStore trait does.

Embeddings live in their own sqlite-vec virtual table, joined to facts by fact_id. FTS5 indexes the same content for keyword search. The hybrid recall pipeline reads both.

06  /  sleep cycle

Eight components,
each does one thing.

Every hour a background worker fires SleepOps. Components are independent — one failing doesn't stop the others.

Five of them call an LLM. Each one has its own temperature + max-tokens: Synthesizer at 0.0 / 512, HandoffWriter at 0.2 / 256, judges at 0.0 with tight budgets.

The expensive ones — CorrectionsAbstractor and ConflictResolver — have their own throttles. Default 24h, settable to 0 to fire every cycle.

The notebook self-cleans

  • 01 Pair upFind schema pairs with cosine similarity ≥ 0.85 — likely about the same topic.

18  /  the closed loop

From correction
to distilled rule.

A single user correction enters the system. Within one session it's recalled. Across multiple sessions it accumulates with siblings. Eventually the CorrectionsAbstractor notices the pattern and promotes them.

The promoted schema outranks the raw corrections (1.6 vs 1.5) — so even though the originals still live in the notebook, the distilled rule is what surfaces first.

If two distilled rules ever disagree, ConflictResolver intervenes: higher confidence wins, recency breaks ties, the loser gets superseded_by set and quietly leaves the recall pool.

Raw feedback → distilled rules → resolved against each other → stale ones retired. The agent gets smarter at remembering what's worth remembering.
