The Problem

Three Memory Organs
That Don't Talk

Before (Passive Storage)

  • Memory checked once at session start, forgotten after
  • Knowledge graph written post-hoc, barely read
  • Subagents arrive with minimal context
  • No mid-execution learning
  • Facts are atoms with no molecular structure
  • Agent re-derives understanding every session

After (Active Cognition)

  • Continuous read/write throughout execution
  • Graph queried by agents on demand
  • Subagents arrive with graph-enriched context
  • Working memory updates every iteration
  • Knowledge compiled into wiki articles
  • Proven procedures recalled and reused

Research Foundation

Standing on the Shoulders of Giants

K

Karpathy's LLM Wiki

Compile knowledge once, don't re-derive on every query. Raw data is source code, wiki is the compiled executable.

M

MAGMA (2026)

Multi-graph memory: semantic, temporal, causal, entity views. Query-adaptive traversal. 45% improvement over baselines.

Z

Zep / Graphiti

Temporal knowledge graphs. Facts change over time. Bitemporal tracking. 90% latency reduction vs MemGPT.

A

A-MEM (NeurIPS 2025)

Self-organizing Zettelkasten memory. Dynamic indexing and linking. Knowledge networks with higher-order attributes.

L

MemGPT / Letta

Two-tier virtual context. Agent manages its own context window like an OS manages virtual memory.

→

Context Engineering

"The delicate art of filling the context window with just the right information for the next step." — Karpathy

06 / 14 Phase 1

Graph as First-Class Citizen

graph_query Tool

Agents can now search entities, explore neighbors, and get contextual subgraphs on demand. Three actions: search, neighbors, context.

Graph-Enriched Delegation

Subagents arrive with entity relationships from the knowledge graph. Entity names extracted from task text, 1-hop neighbors appended.

Temporal Facts

Facts track when they became true and when superseded. Outdated facts get 0.3x scoring penalty. Contradictions auto-detected.

Agent calls graph_query: > graph_query(action="neighbors", query="yfinance") ## Neighbors of "yfinance" (both, depth=1) - yfinance --> rate_limiting (pattern) via `requires` - yfinance --> quote_data (function) via `provides` - yfinance <-- pandas (module) via `integrates_with`

08 / 14 Phase 3

Ward Wiki — Karpathy Compiler

Knowledge is compiled once, not re-derived on every query. Each ward accumulates a structured wiki that grows across sessions.

Without Wiki (Session 50)

  • Recall 5 flat facts from 500+ stored
  • Re-discover rate limiting patterns
  • Re-learn pandas quirks
  • Same mistakes, different session

With Wiki (Session 50)

  • Load wiki index, pull 3 relevant articles
  • "yfinance-patterns" article: 400 words of compiled knowledge
  • "rate-limiting-strategies" article: proven approaches
  • Knowledge compounds, session 51 is dramatically better
Session Ends → Distillation → Wiki Compilation → LLM: "Update wiki with new facts" → Upsert Articles
09 / 14 Phase 4

Procedural Memory — Learning HOW

The agent remembers not just facts but proven multi-step action sequences. When a similar task appears, the procedure is suggested during intent analysis.

Extracted Procedure: "stock_analysis_report" Steps: 1. ward → enter "stock-analysis" 2. delegate → research-agent: "fetch {ticker} data" 3. delegate → research-agent: "analyze options chain" 4. delegate → writing-agent: "create HTML dashboard" 5. respond → markdown with artifacts Parameters: [ticker] Success rate: 87% (7 successes / 1 failure) Trigger: "stock analysis", "portfolio report", "market dashboard"

Complete Pipeline

Session Lifecycle — New vs Continuation

NEW SESSION User Message → Recall (wiki articles + facts + graph + procedures) → Intent Analysis (with procedure matching) → Execute → Working Memory updates each iteration → Micro-recall at tool calls, errors, delegation → Graph queries by agent on demand → Distill → Extract facts, entities, relationships → Extract procedures (if successful) → Compile ward wiki articles → Update temporal facts (supersession) CONTINUATION Follow-up Message → Recall (same as above) → Intent Analysis SKIPPED → Execute (full history loaded, WM active) → Distill (same as above)

Live Example

User asks: "Analyze SPY stock"

1. Session Start Recall

Wiki article "yfinance-patterns" loaded (compiled from 12 prior sessions). Graph shows SPY → requires → rate_limiting. Procedure "stock_analysis_report" recalled (87% success rate).

2. Intent + Procedure

Intent analysis sees the proven procedure. Recommends: delegate to research-agent, then writing-agent. Ward: "stock-analysis".

3. Execution with WM

Working memory tracks SPY entity. When shell returns an error ("rate limit exceeded"), micro-recall fires: "Use 1 req/sec delay" correction injected. Agent self-corrects immediately.

4. Post-Session

Distillation extracts new facts. Wiki article updated. Procedure success_count incremented. Old fact "SPY at $510" superseded by "SPY at $523".
